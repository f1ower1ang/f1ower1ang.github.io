用effect同步

effect可以在组件渲染后运行一些代码，使得组件与React之外的系统同步。

它和事件处理器的区别是使用effect可以指定由渲染本身而非特定事件引起的副作用。

effect使用方法

使用effect主要分为三步：

  • 声明一个effect。effect默认在每次渲染完后执行。

    • import { useEffect } from 'react'
function MyComponent() {
  useEffect(() => {
    // 每次渲染后都会执行这里面的代码
  })
  return <div />;
}

  • 按需返回清理函数。一些effect需要在下次effect执行前完成暂停、取消或清理工作。

    • 例如，网络连接建立后，需要在组件卸载前断开连接。

开发环境下React会执行两次effect

在开发环境下，React故意重新挂载两次组件，这有利于我们发现一些潜在的bug。因此，在编写组件时，应该关注组件在这两次挂载时的变化，避免报错。

在生产环境下，React组件的挂载按照正常流程处理，即只挂载一次。

响应式effect的生命周期

effect与组件的生命周期不同，组件经历挂载、更新或卸载；而effect只做两件事：依赖变化时执行effect，之后执行清理函数，这个流程会根据effect依赖的props和状态改变执行多次，确保effect同步的是最新的props和状态。

一个effect的生命周期

每个React组件会经历相同的声明周期：

  • 当组件被添加到屏幕时会挂载
  • 当组件接收新的props或者状态时会更新
  • 当组件被移除时会卸载

每个effect的生命周期需要独立于组件来思考，可以通过一个和服务器建立连接的例子来感受一下。

假设ChatRoom组件有一下流程：

  • ChatRoom挂载时，给roomId设置为”general”
  • 给roomId设置”travel”伴随着ChatRoom组件更新
  • 给roomId设置”music”伴随着ChatRoom组件更新
  • ChatRoom卸载

在组件的每个生命周期节点中，effect内部执行流程不一样：

  • effect创建”general”房间的连接
  • effect断开与”general”房间的连接然后建立”travel”房间的连接
  • effect断开与”travel”房间的连接然后简历”music”房间的连接
  • effect断开”music”房间的连接

在思考effect的生命周期时，只需要专注于每次的开始/暂停，不应该关注组件是否处于挂载、更新或卸载状态。唯一需要处理的逻辑是如何开始同步React外的信息以及如何暂停，这样才能使得更加effect灵活。

effect的依赖为空

稍微修改一下房间连接建立的例子，使得effect的依赖为空。

当前effect不使用任何响应值，因此依赖为空。从组件的角度来思考，当组件挂载并渲染完会执行一次effect，即建立连接，当组件卸载时断开连接。

如果从effect的角度来考虑，不需要关注挂载和卸载，而是关注effect中同步逻辑的开始和暂停。
